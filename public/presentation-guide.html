Sesi 2 — Konsep Solusi: SHA-256 (1.5 Menit)

Apa yang Disampaikan:

  • "DocuVerify UNY menggunakan kriptografi Hash SHA-256. Alih-alih menyimpan file PDF ke database, sistem hanya menciptakan 'Sidik Jari Digital' dari file tersebut."
  • "Hash bersifat satu arah (one-way function) — tidak bisa dibalik menjadi isi dokumen asli, sehingga privasi tetap terjaga."
  • "Hash juga bersifat unik — jika ada satu huruf atau satu titik saja yang ditambahkan atau dihapus di PDF, kode Hash-nya akan berubah total (100%)."
💡 Contoh untuk Dosen: File PDF asli → Hash: a7f3c9d8e1...
File PDF diedit 1 huruf → Hash: 5b2e4f0a8c... (berubah total!)

Sesi 3 — Teknologi yang Digunakan (1.5 Menit)

Apa yang Disampaikan:

"Berikut adalah tech stack utama yang saya gunakan untuk membangun website ini:"

N

Next.js 15 (React)

Kategori: Framework Fullstack

Framework React terbaru untuk membangun frontend dan backend (API Routes) dalam satu project. Mendukung Server-Side Rendering (SSR) dan Static Generation untuk performa maksimal.

S

Supabase (PostgreSQL)

Kategori: Database Cloud

Database relasional PostgreSQL yang di-hosting di cloud. Menyimpan metadata dokumen dan kode Hash SHA-256. Diakses melalui koneksi pooler (PgBouncer) agar efisien dan scalable.

P

Prisma ORM

Kategori: Object-Relational Mapping

Library untuk berinteraksi dengan database menggunakan TypeScript. Menyederhanakan query database menjadi kode yang aman secara tipe data (type-safe).

T

TypeScript

Kategori: Bahasa Pemrograman

Superset dari JavaScript yang menambahkan sistem tipe statis. Mengurangi bug dan meningkatkan kualitas kode secara keseluruhan.

W

Web Crypto API (SHA-256)

Kategori: Kriptografi

API bawaan browser modern untuk menghitung Hash SHA-256 dari file PDF. Proses hashing dilakukan di sisi client (browser), sehingga file tidak perlu dikirim ke server untuk dihash.

J

JWT + bcrypt

Kategori: Autentikasi & Keamanan

JSON Web Token (JWT) untuk manajemen sesi admin. bcrypt untuk hashing password. Dilengkapi CSRF protection dan Rate Limiting untuk mencegah serangan brute-force.

Arsitektur Singkat:

🏗️ Arsitektur Aplikasi Frontend (React/Next.js) → API Routes (Next.js Backend) → ORM (Prisma) → Database (Supabase PostgreSQL)

Semua berjalan dalam satu project Next.js (monolith fullstack), sehingga deployment cukup satu kali ke platform seperti Vercel.

Sesi 4 — Alur Sistem: Pendaftaran Dokumen (1.5 Menit)

Apa yang Disampaikan:

"Berikut alur sistem saat pihak kampus (Admin) mendaftarkan dokumen baru:"

1
Otentikasi Admin — Admin melakukan login menggunakan email dan password. Sistem memvalidasi kredensial menggunakan bcrypt, membuat sesi JWT, dan menyimpannya sebagai HTTP-Only Cookie. Dilindungi oleh Rate Limiting (maks 5 percobaan/10 menit).
↓
2
Input Metadata — Admin mengisi formulir: Nama Pemilik, NIM, Fakultas, Program Studi, Jenis Dokumen (Ijazah/Sertifikat/Transkrip), dan Tahun.
↓
3
Upload & Ekstraksi Hash — Admin mengunggah file PDF asli. File tersebut tidak disimpan ke cloud storage. Sistem langsung membaca konten biner file dan menghitungnya menjadi kode Hash SHA-256 (contoh: e3b0c44298fc1c14...).
↓
4
Simpan ke Database — Sistem mengirimkan data teks (metadata + kode Hash) ke database PostgreSQL via Prisma ORM. File PDF asli tidak ikut tersimpan.
↓
✓
Selesai — Dokumen kini terdaftar secara sah. Audit log dicatat untuk keperluan jejak aktivitas admin.

Sesi 5 — Alur Sistem: Verifikasi Publik (1.5 Menit)

Apa yang Disampaikan:

"Sekarang, alur saat pihak luar (HRD/Perusahaan) memverifikasi keaslian dokumen:"

1
Akses Publik — Verifikator membuka halaman utama website. Tidak perlu login. Halaman ini dirancang agar siapa saja bisa memverifikasi dokumen.
↓
2
Upload Dokumen — Verifikator mengunggah file PDF sertifikat/ijazah yang diterimanya (misalnya melalui Email atau WhatsApp dari pelamar kerja). Validasi dilakukan: hanya file PDF dan maks 10 MB.
↓
3
Kalkulasi Hash — Sistem secara instan menghitung ulang kode Hash SHA-256 dari file PDF yang baru diunggah tersebut menggunakan Web Crypto API.
↓
4
Pencocokan Database (Query) — Kode Hash hasil perhitungan tadi dikirim ke API Backend, yang kemudian melakukan query ke database: SELECT * FROM Document WHERE fileHash = '...'
↓
5
Keputusan Validasi:
✅
VALID (Hash Cocok)
Database menemukan record yang cocok. Layar berubah HIJAU. Metadata asli (Nama, NIM, Fakultas) ditampilkan sebagai bukti keaslian.
❌
TIDAK TERDAFTAR (Hash Tidak Cocok)
Hash tidak ditemukan di database. Artinya dokumen belum didaftarkan ATAU isinya sudah diedit/dipalsukan. Layar berubah MERAH.

Sesi 6 — Live Demo Website (2 Menit)

Langkah Demo:

A. Demo Pendaftaran (Sisi Admin):

  1. Buka http://localhost:3000/admin/login, login dengan akun admin.
  2. Masuk ke Dashboard, tunjukkan ringkasan statistik.
  3. Klik "Tambah Dokumen". Isi metadata, unggah Sertifikat_Asli.pdf. Klik Simpan.
  4. Jelaskan: "Sistem baru saja menghitung Hash SHA-256 dari file ini dan menyimpannya ke database."

B. Demo Verifikasi Berhasil:

  1. Buka tab baru (halaman publik) http://localhost:3000/verify.
  2. Unggah Sertifikat_Asli.pdf yang sama.
  3. Hasil: Layar HIJAU (VALID) + data metadata muncul.

C. Demo Penolakan (Pembuktian Anti-Pemalsuan):

  1. Unggah Sertifikat_Palsu.pdf (file asli yang sudah diedit sedikit isinya).
  2. Hasil: Layar MERAH (TIDAK TERDAFTAR).
  3. Jelaskan: "Cukup mengubah 1 huruf saja maka Hash-nya berubah total, dan sistem langsung menolaknya."
⚠️ Persiapan Sebelum Demo: Siapkan 2 file di Desktop Anda:
1. Sertifikat_Asli.pdf — Untuk didaftarkan dan lolos verifikasi.
2. Sertifikat_Palsu.pdf — File asli yang sudah Anda ubah 1 kata menggunakan PDF editor, lalu save ulang.

💡 Tips Menghadapi Pertanyaan Dosen

Q: "Kenapa tidak simpan saja file PDF-nya di database?" A: "Menyimpan file biner PDF sangat menghabiskan storage dan berisiko terhadap kebocoran data pribadi. Dengan hashing, kita hanya menyimpan teks 64 karakter, namun tetap bisa membuktikan keaslian dokumen."
Q: "Apa kelebihan SHA-256 dibanding metode lain?" A: "SHA-256 menghasilkan hash 256-bit yang secara statistik impossible menghasilkan collision (dua file berbeda yang memiliki hash sama). Ini standar yang digunakan oleh Bitcoin dan berbagai sistem keamanan dunia."
Q: "Bagaimana jika dokumen dikirim via WhatsApp?" A: "Selama dikirim sebagai 'Dokumen' (bukan sebagai gambar/foto), file PDF tidak berubah dan akan tetap terverifikasi Valid karena data binernya identik."
Q: "Kenapa memilih Next.js?" A: "Next.js adalah framework fullstack — frontend dan backend (API) dalam satu project. Mendukung SSR untuk performa, dan sangat mudah di-deploy ke platform seperti Vercel."
